# Spec Workflow MCP — Claude-Led Collaborative Development Framework

[中文文档](README.zh.md)

Claude-led collaborative development framework. Claude Code plans, implements, and reviews; OpenAI Codex is an optional engine you offload specific coding tasks to. Built on [Pimzino/spec-workflow-mcp](https://github.com/Pimzino/spec-workflow-mcp), with traffic-light verification and academic report generation.

> **Note**: Files in `docs/` describe upstream Pimzino behavior. For fork-specific usage, see the project `CLAUDE.md` generated by `init.sh`.

## Installation

### Step 1: Clone and build

```bash
git clone https://github.com/sana3419/spec-workflow-mcp.git
cd spec-workflow-mcp
npm install && npm run build
```

### Step 2: Install the Codex CLI

```bash
npm i -g @openai/codex
codex login    # First run: sign in with your ChatGPT account (or configure an OpenAI API key)
```

### Step 3: Initialize your project

```bash
bash /path/to/spec-workflow-mcp/templates/init.sh /path/to/your-project
```

Optional flags: `--with-graph` (code-review-graph), `--with-nexus` (GitNexus), `--with-all` (graph + nexus), `--auto-loop` (pre-enable the Phase 4 background loop), `--force` (overwrite existing).

> Want codebase visualization? [Understand-Anything](https://github.com/Lum1104/Understand-Anything) is a separate Claude Code plugin (not managed by `init.sh`) — install it from inside Claude Code with `/plugin install understand-anything`.

### Step 4: Start

```bash
cd /path/to/your-project && claude
```

First time: approve the MCP servers when prompted.

## How It Works

```
Phase 1-3  Claude plans (Requirements → Design → Tasks, each confirmed in conversation)
Phase 4    Execute tasks (spec-status → Claude implements by default; offload to Codex only for _Engine: codex tasks → verify-task → log-implementation)
Phase 5    Research report (optional: Markdown + SVG → rsvg-convert → gen-report.py → docx)
```

**Approval is in-conversation.** Claude presents each `requirements.md` / `design.md` / `tasks.md` to you in the chat; the next phase begins only after you approve it in the conversation. There is no Dashboard approval page and no VS Code approval flow.

## Codex Dispatch (optional)

By default Claude implements each task itself. When a task is tagged `_Engine: codex` in `tasks.md` — useful for large, repetitive, or parallelizable coding, or to save Claude's context — Claude offloads that task to OpenAI Codex through Codex's **own MCP server**. Claude never calls CLIs directly. `init.sh` writes the codex server to the project `.mcp.json`:

```json
"codex": { "type": "stdio", "command": "codex", "args": ["mcp-server"], "env": {} }
```

The codex MCP server exposes two tools:

- `mcp__codex__codex(prompt, sandbox, approval-policy, model, cwd)` → starts a new Codex session, returns a `threadId`
- `mcp__codex__codex-reply(threadId, prompt)` → continues an existing session

**Per-spec session reuse.** Each spec stores one threadId in `.spec-workflow/specs/<spec>/.codex-thread`. The first task of a spec calls `codex()` and saves the threadId; later tasks in the same spec and any red→fix retry call `codex-reply()`. A new session is started only for a new/unrelated spec, or if `codex-reply` fails (stale thread). This keeps the worker's context across a spec's tasks and during fixes — efficient and accurate.

**Codex knobs** come from `.spec-workflow/config.toml` `[engine.codex]`:

| Key | Values | Default |
|-----|--------|---------|
| `sandbox` | `read-only` \| `workspace-write` \| `danger-full-access` | `workspace-write` |
| `approvalPolicy` | `untrusted` \| `on-failure` \| `on-request` \| `never` | `never` |
| `model` | (optional) Codex model | — |

There is also `[engine] default = "claude"` and `maxFixAttempts = 5`. The full file:

```toml
[engine]
default = "claude"          # claude | codex (claude = Claude implements directly; codex = offload to Codex)
maxFixAttempts = 5           # max red→fix retries before a task is auto-blocked

[engine.codex]
sandbox = "workspace-write"  # read-only | workspace-write | danger-full-access
approvalPolicy = "never"     # untrusted | on-failure | on-request | never
# model = "..."             # optional — leave commented out to use Codex's latest default (recommended)

[loop]
autoLoop = false             # master on/off for the background loop runner
maxIterations = 50           # hard cap on auto-loop iterations
noProgressStop = 3           # stop after N iterations with no tasks.md change
```

**Autonomous verify→fix loop.** dispatch → worker writes a report to `.spec-workflow/reports/codex-<task>-<timestamp>.md` → Claude runs tests → `verify-task` green/red → on red, `codex-reply` with the failing log (bounded by `maxFixAttempts`) → `log-implementation`.

## Background Auto-Loop (optional)

By default you drive Phase 4 in your interactive session. To run it **hands-off**, start the background loop runner — it drives the spec to completion in a **separate, headless `claude` process**, so your interactive session **stays free to chat and check progress**.

`init.sh` installs the runner at `.spec-workflow/spec-loop-run.sh`. Enable + start it:

```bash
# 1. enable in .spec-workflow/config.toml
[loop]
autoLoop = true          # master on/off (or pass --auto-loop to init.sh, or just ask Claude)

# 2. start it in the background (from the project root)
nohup bash .spec-workflow/spec-loop-run.sh <spec-name> >/dev/null 2>&1 &
```

You can also just tell Claude **"run the loop in the background"** and it launches the runner for you, then keeps chatting.

- **Watch progress:** `tail -f .spec-workflow/loop-run.log`, or `spec-status`, or the dashboard — all while your session stays interactive.
- **Stop:** `touch .spec-workflow/.loop-stop` (or `kill "$(cat .spec-workflow/.loop-run.pid)"`).
- **Guardrails:** `maxIterations` (default 50) hard-caps it; `noProgressStop` (default 3) stops after N iterations with no `tasks.md`/`verify-results` change. Every iteration + stop reason is logged to `.spec-workflow/loop-audit.log`.
- **How it works:** the runner invokes a fresh headless `claude` per task (implement → test → verify-task → log-implementation), reading shared state from disk — classic "agentic loop" pattern.

## MCP Architecture

```
Claude Code (orchestrator)
├── spec-workflow-mcp     — workflow management (5 tools)
├── codex (MCP server)    — coding dispatch
│   ├── codex(prompt, sandbox, approval-policy, model)  → new session (threadId)
│   └── codex-reply(threadId, prompt)                   → reuse session
├── code-review-graph     — knowledge graph (optional, --with-graph)
└── gitnexus              — dependency analysis (optional, --with-nexus)
```

## MCP Tools

| Server | Tool | Purpose |
|--------|------|---------|
| spec-workflow-mcp | `spec-workflow-guide` | Full workflow guide (call first each session) |
| | `steering-guide` | Project steering documents |
| | `spec-status` | Progress + next-task dispatch hint |
| | `verify-task` | Traffic-light verification (green→done, red→fix/blocked) |
| | `log-implementation` | Record implementation logs and artifacts |
| codex | `mcp__codex__codex` | Start a new Codex session (returns threadId) |
| | `mcp__codex__codex-reply` | Continue an existing Codex session by threadId |

## Using Claude Code

```
"Build a user auth system"       → spec-workflow planning
"Show progress"                  → spec-status
"Tests passed / failed"          → verify-task green/red
"Use /review"                    → launches 4 review subagents (Claude)
"Use /tdd to develop this"       → dispatches Codex for TDD
```

## Skills (auto-installed to .claude/skills/)

| Skill | Engine | Purpose |
|-------|--------|---------|
| `/review` | Claude | Launches 4 review subagents in parallel (security, logic, performance, API) |
| `/qa` | Codex | Systematic QA testing + atomic fixes |
| `/design-review` | Claude | Visual/interaction audit (multimodal) |
| `/tdd` | Codex | TDD red-green-refactor + worktree isolation |

## Subagents (auto-installed to .claude/agents/)

| Agent | Focus |
|-------|-------|
| `security-reviewer` | Injection, auth flaws, hardcoded secrets, CVE |
| `logic-reviewer` | Edge cases, race conditions, resource leaks |
| `performance-reviewer` | N+1 queries, memory leaks, blocking ops |
| `api-reviewer` | Naming, HTTP semantics, versioning, validation |

## Statusline

Auto-deployed to `~/.claude/statusline.sh`. Shows: model, context %, tokens, cost, duration, code changes. Auto-saves session usage to `.spec-workflow/session-usage.json`.

## Project Structure (after init)

```
your-project/
├── CLAUDE.md                          # Workflow guide (auto-read by Claude Code)
├── AGENTS.md                          # Codex worker instructions (read natively by Codex)
├── .mcp.json                          # MCP server config
├── .claude/
│   ├── settings.json                  # Permission config
│   ├── agents/                        # Review subagents
│   └── skills/                        # /review, /qa, /design-review, /tdd
└── .spec-workflow/
    ├── config.toml                    # Engine + loop config
    ├── steering/                      # Project steering (optional)
    ├── specs/                         # Spec documents
    │   └── <spec>/                    # requirements/design/tasks.md
    │       ├── .codex-thread          # Per-spec Codex session id
    │       ├── Implementation Logs/   # log-implementation artifacts
    │       └── verify-results/        # verify-task results
    ├── reports/                       # Codex output reports (codex-<task>-<timestamp>.md)
    ├── usage-log.json                 # Per-task usage (engine, tokens, cost)
    └── session-usage.json             # Per-session usage (model, tokens, lines changed)
```

## Troubleshooting

| Problem | Solution |
|---------|----------|
| MCP tools not available | Restart Claude Code. First time: approve MCP servers when prompted |
| Codex dispatch fails | Run `codex login`. Check `codex --version`. Check `.mcp.json` has the codex server |
| Dashboard won't start | `lsof -i :5000` to check port |
| Reset MCP server trust | `claude mcp reset-project-choices` |
